# CosMate 標籤體系 × 推播設計｜簡報大綱（18 頁）

> 目的：向**不熟產品的同事與工程師**說明——為什麼要做標籤體系、長怎樣、先做哪一段
> 結構邏輯：案例先建心理圖像 → 架構補完整骨架 → 分階段落地 + 假說驗證
> 建議時長：20–25 分鐘（含 Q&A 30 分鐘）

## 第一段：開場（2 頁）

### P1 — 封面
- 標題：**CosMate 標籤體系｜從亂槍打鳥到精準協商**
- 副標：三層標籤 × 四條推播原則 × 分階段落地
- 講者 / 日期 / 版本

### P2 — 為什麼此刻要做（1 頁 Summary）
- **現狀一句話**：CosMate 留存率低，用戶常在「加好友」節點流去 IG / Discord
- **推播現狀**：目前是廣撒網，對潛水用戶更是反效果——愈推愈安靜
- **本次提案**：用**標籤**描述用戶狀態、用**權重**跟隨行為漲跌、用**推播**做有禮貌的協商
- **今天要你帶走的 3 件事**：
  1. 標籤不是貼紙，是隨行為漲跌的權重向量
  2. 推播不是觸達，是對用戶邊界的持續協商
  3. 第一階段只做一件事：**喚活潛水用戶**

## 第二段：案例先行（5 頁，建立心理圖像）

> 設計原則：每頁一個虛構但貼近真實的用戶，讓不熟產品的人「先看到人，再看到系統」。每頁版面固定：**左側人物卡｜右側行為時間線 + 推播設計**。

### P3 — 案例 A：潛水 14 天的原神 Coser 小安（喚活情境）
- **基本**：25 歲 / 台北 / 女性 Coser / 註冊 6 個月
- **近期行為**：最後發文 14 天前（原神新角 Cos 照）、App 開啟頻率從週 3 降到 0、未讀訊息 12 則
- **傳統推播（錯誤示範）**：「3 位新朋友想認識你！」→ 被滑掉 / 封鎖通知
- **標籤推播（正確）**：
  - 觸發條件：潛水 > 7 天 + 原神標籤權重 > 0.6 + introvert_mode = 0
  - 推播文案：「原神 5.3 新角你最想抽哪個？」（輕量投票，不需開對話）
  - 頻率上限：週 1 則
- **學到的事**：對潛水用戶，**降低互動成本** > 增加觸達次數

### P4 — 案例 B：註冊 3 天、還在觀望的阿翔（破冰情境）
- **基本**：19 歲 / 香港 / 男性 / 動漫愛好者、非 Coser
- **近期行為**：註冊完填了 3 個 IP 標籤（鏈鋸人 / 咒術迴戰 / 間諜家家酒）、瀏覽 4 個檔案、0 對話
- **痛點**：新用戶不知道怎麼開口，默默關 App
- **標籤推播設計**：
  - 觸發：註冊 ≤ 7 天 + 有 IP 標籤 + 0 對話
  - 推薦：每日破冰話題（該 IP 最新話題）+ 「今天有 3 位也喜歡鏈鋸人的人在線」
  - 特殊：前 7 天不套疲勞度上限（冷啟動寬容期）
- **學到的事**：新用戶要的是**被引導**，不是被配對

### P5 — 案例 C：週週來但不說話的攝影師 Mika（輕量互動情境）
- **基本**：30 歲 / 日本 / Coser 攝影師 / 重度瀏覽型用戶
- **近期行為**：週 5 次 App 開啟、平均每次 8 分鐘瀏覽作品集、按愛心 > 30 次、主動對話 = 0
- **標籤畫像**：`深度瀏覽型` × `愛按愛心` × `不主動開話題`
- **標籤推播設計**：
  - 不推「想認識新朋友嗎？」（對這種人無效）
  - 推「你按過愛心的 @xxx 剛發了新作品」（延續既有行為軌跡）
  - 推「這週你欣賞過 12 位創作者，他們的共同愛好是……」（讓用戶感覺被理解）
- **學到的事**：沉默≠沒興趣，標籤要能辨識**非對話型的參與**

### P6 — 案例 D：揪團傾向高但最近沒活動的 Leo（活動曆情境）
- **基本**：27 歲 / 高雄 / Coser / 曾揪過 2 次 FF 場
- **近期行為**：最近一次揪團 3 個月前、最近點開活動曆 5 次但未發揪團
- **痛點**：想揪沒動力、怕發了沒人理
- **標籤推播設計**：
  - 觸發：`揪團傾向` ≥ 0.7 + 近 7 天點過活動曆 + 地區匹配
  - 推播：「FF43 有 8 位同好已加入追蹤，要不要開個揪團？」（給社會證據降低發起門檻）
- **學到的事**：揪團推播的關鍵是**讓用戶知道自己不會孤單**

### P7 — 四個案例的共同模式（轉場頁）
- **提取**：
  1. 同一個 App，4 種人需要 4 種推播邏輯 → **必須有結構化畫像**
  2. 關鍵訊號不只「喜歡什麼 IP」，更重要是「現在的行為狀態」
  3. 推播成敗的分水嶺是：**有沒有尊重當下的邊界**（睡眠/忙碌/觀望）
- **承上啟下**：下一段說明，標籤如何**同時描述「是誰」「喜歡什麼」「現在在哪種狀態」**

## 第三段：架構展開（6 頁）

### P8 — 三層標籤總覽（一張圖）
- 視覺：三層疊圖 + 更新頻率軸
- 重點：**靜態 → 半靜態 → 動態**，三層組合出畫像
- 一句話：「Layer 1 說你是誰，Layer 2 說你喜歡什麼，Layer 3 說你現在在哪」

### P9 — Layer 1：基礎屬性標籤（低頻 / 註冊即定）
- 來源：註冊填寫、個人檔案
- 例：性別 / 年齡層 / 地區（台港澳日）/ 主要身份（Coser / 攝影師 / 純粉絲）
- 用途：配對池粗篩、活動曆地區過濾
- 工程重點：`cosmate_users` 主檔欄位，用戶可編輯

### P10 — Layer 2：興趣 IP 標籤（中頻 / 權重版）
- 資料結構：`tag_id + weight(0-1) + last_active_at + source`
- 來源：用戶勾選 + 作品集辨識 + 聊天關鍵字
- 權重怎麼動：互動行為 → 微調；長期沉默 → 衰減
- 例：`原神(0.8) × 鳴潮(0.6) × 崩鐵(0.3)`
- 工程重點：新表 `user_interest_tags`，事件驅動更新

### P11 — Layer 3：行為狀態標籤（高頻 / 即時）
- 來源：App 內行為事件流
- 例：`輕量互動/愛投票`、`深度瀏覽型`、`潛水型`、`週末活躍`、`揪團傾向`
- 更新頻率：事件即時 or 每日 batch
- 用途：**這一層是推播節奏的主開關**
- 工程重點：事件隊列 → 規則引擎 → 標籤權重更新

### P12 — 推播四條過濾原則（原檔 4 點落成 API 規則）
1. **內向模式硬排除**：`introvert_mode = 1` 不入推播名單
2. **反饋迴圈降權**：連續 dismiss → 自動降 Layer 3 權重
3. **疲勞度上限**：潛水用戶週 1–2 則、無互動則拔標籤
4. **狀態絕對過濾**：`account_status = active` API 閘道強制檢查

> 這四條是**推播的憲法**，任何推播邏輯先過這四關

### P13 — 跨層交叉：一條完整的推播決策鏈
- 用案例 A（小安）走一次完整決策：
  1. Layer 1 命中：台灣 × 女 × Coser
  2. Layer 2 命中：原神(0.8)
  3. Layer 3 命中：潛水型（14 天）× 輕量互動(0.7)
  4. 四條過濾：introvert_mode=0 ✅、狀態=active ✅、本週已推 0/1 ✅
  5. 觸發：原神新角投票推播
  6. 結果反饋：若 dismiss → 投票權重 -0.1、若點擊 → +0.05
- 這張圖同時展示**標籤如何驅動推播、推播結果如何回寫標籤**（閉環）

## 第四段：分階段落地（5 頁）

### P14 — 現況與最迫切需求（診斷頁）
- **最大痛點**：留存低，潛水用戶佔比 __%（待補數據）
- **工程資源現狀**：__ 位工程師 / 每週 __ 人日可投入
- **選擇的邏輯**：不做全量標籤 = 先做**回本最快**的一段
  - 潛水用戶是最大流失點 → 先做喚活
  - 新用戶破冰是第二痛點 → Phase 2
  - 其他（Mika / Leo 情境）→ Phase 3
- **這頁要達成**：讓工程師看到「我們不是要你蓋整個帝國，只蓋第一棟」

### P15 — Phase 1：潛水用戶喚活 MVP（4 週）
| 週 | 產出 | 負責 |
|---|---|---|
| W1 | `cosmate_tags` schema + DDL（Layer 1 + 簡化版 Layer 3） | 工程鴿頭子 |
| W2 | 潛水判定規則 + 4 條過濾規則 API | 工程鴿頭子 |
| W3 | 投票推播模板 × 3（主要 IP）+ 權重更新函式 | COCO 鴿 |
| W4 | 1000 人 A/B 實驗上線 | PM + 工程 |

- **範圍紀律**：Phase 1 **不做** Layer 2 完整版、不做新用戶破冰、不做作品集辨識
- **Phase 1 成功標準**：A/B 潛水用戶 7 日回流率 +20%

### P16 — Phase 2 / Phase 3 路徑圖（可視化但不展開細節）
- **Phase 2**（第 5–8 週）：新用戶破冰（案例 B）+ 完整 Layer 2 IP 標籤體系
- **Phase 3**（第 9–12 週）：深度瀏覽型推播（案例 C）+ 揪團推播（案例 D）
- **關鍵判斷點**：Phase 1 若未達 +20%，**不進 Phase 2**，先修模型
- 這頁告訴團隊：**每一階段都有退場機制**，不是一路衝到底

### P17 — 假說驗證設計（A/B + 指標）
- **核心假說**：「基於 Layer 3 行為狀態的推播，對潛水用戶的喚活效果優於廣撒式推播」
- **實驗設計**：
  - 對照組：現行廣撒式推播
  - 實驗組：Phase 1 標籤推播
  - 樣本：潛水 ≥ 7 天的用戶各 500
  - 週期：4 週
- **主要指標**：7 日內回流率
- **護欄指標**：推播封鎖率、關通知率、應用移除率（不得惡化 > 2%）
- **次要指標**：推播打開率、投票完成率、後續對話發起率
- **驗證不過的處理**：檢查是 Layer 3 判定錯、文案錯、還是時機錯——不是直接放棄

### P18 — 風險、未決問題、Q&A
- **風險**：
  1. Layer 3 冷啟動期誤判新用戶為潛水
  2. 標籤權重更新頻率 vs DB 負載
  3. 推播文案多樣性（3 個模板會不會膩）
- **未決問題（要工程師幫忙想）**：
  - 權重更新走即時 trigger 還是每日 batch？
  - 標籤表放主庫還是獨立 store？
  - A/B 分組在哪一層實作？
- **下一步**：
  1. 工程鴿頭子出 schema 初稿（W1）
  2. PM 補齊現況數據（潛水用戶佔比、留存基線）
  3. 收集本次簡報的 3 個最尖銳質疑，下週回覆

## 給講者的小抄

- **節奏建議**：案例段不要趕，每個案例給聽眾 30 秒消化；架構段可以加速
- **最容易被質疑的點**：
  1. 「這跟現有推播差在哪？」→ 用 P13 決策鏈回答
  2. 「Phase 1 為什麼不一次做完？」→ 用 P14 回本最快邏輯回答
  3. 「標籤會不會越長越多失控？」→ 答：有退場機制（疲勞度 + 反饋迴圈拔標籤）
- **不要說的話**：不要講「標籤是權重向量」這種內部哲學術語，聽眾聽不懂
- **要說的話**：多講「尊重邊界」「降低互動成本」「有禮貌的協商」這種人話
