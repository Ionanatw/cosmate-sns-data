## 一、摘要（Source Doc 濃縮）

原檔只涵蓋**投票推播對潛水用戶的 4 條設計原則**，並未完整說明標籤體系全貌。4 條重點：

| # | 設計原則 | 底層欄位/機制 | 用戶邊界尊重 |
|---|---------|--------------|-------------|
| 1 | 內向模式開關 | `cosmate_users.introvert_mode` | 硬性排除，不踩邊界 |
| 2 | 反饋迴圈 | `action_type`（dismiss/ignore）→ 自動降權 | 行為即訊號 |
| 3 | 疲勞度控管 | 潛水用戶每週 1-2 則上限；長期無互動 → 拔標籤 | 動態退場機制 |
| 4 | 絕對狀態過濾 | `account_status = active` | API 閘道強制檢查 |

**核心設計哲學**：標籤不是靜態貼紙，是**隨行為漲跌的權重向量**；推播不是觸達，是**對邊界的持續協商**。

## 二、標籤體系架構（Proposed）

原檔只談推播過濾，未定義標籤本體。以下補完三層架構，讓推播設計有完整底座。

### Layer 1：基礎屬性標籤（靜態 / 低頻更新）
- **來源**：註冊填寫、個人檔案
- **例**：性別、年齡層、地區（台/港/澳/日）、主要活躍身份（Coser / 攝影師 / 純粉絲）
- **更新頻率**：用戶主動編輯時
- **用途**：粗篩配對池、活動曆地區過濾

### Layer 2：興趣 IP 標籤（半靜態 / 中頻更新）
- **來源**：用戶勾選 + 作品集辨識 + 聊天關鍵字
- **例**：原神 / 崩鐵 / 鏈鋸人 / 初音 / 特攝 …
- **結構**：`tag_id + weight(0-1) + last_active_at + source`
- **更新頻率**：每次互動行為觸發權重微調
- **用途**：破冰話題推薦、同好配對、IP 活動通知

### Layer 3：行為狀態標籤（動態 / 高頻更新）
- **來源**：App 內行為事件流
- **例**：`輕量互動/愛投票`、`深度聊天型`、`潛水型`、`週末活躍`、`活動揪團傾向`
- **更新頻率**：事件即時 or 每日 batch
- **用途**：推播節奏控制、喚活時機判斷、留存預警

### 三層交叉範例（潛水用戶喚活）
```
Layer 1: 台灣 × 女性 × Coser
Layer 2: 原神(0.8) × 鳴潮(0.6) × 崩鐵(0.3)
Layer 3: 潛水型(連續 14 天未發文) × 輕量互動/愛投票(0.7)
→ 觸發：「原神新角你選誰？」輕量投票推播（週 1 則上限）
→ 若連 3 次 dismissed：降 Layer 3 投票權重 → 拔除該標籤
```

## 三、執行規劃（4 週 MVP）

| 週 | 里程碑 | 產出物 | 負責 |
|---|--------|--------|------|
| W1 | 資料模型定稿 | `cosmate_tags` schema + 3 張關聯表 DDL | 工程鴿頭子 |
| W2 | 標籤事件管線 | App 行為事件 → 權重更新函式（純 SQL / Cloudflare Workers） | COCO鴿 |
| W3 | 推播 API 改造 | 4 條過濾規則落 API 閘道（introvert / feedback / fatigue / status） | 工程鴿頭子 |
| W4 | A/B 驗證 | 潛水用戶 1000 人分組實驗：舊推播 vs 新標籤推播，觀察 7 日回流率 | PM大大鴿 |

### 驗收指標
- 潛水用戶推播打開率：baseline 對照組 + 20% 為成功
- 推播封鎖/關通知率：不惡化（< baseline + 2%）
- 拔標籤觸發次數：每週 > 0（代表動態機制有在跑）

### 風險與反問
1. **Layer 3 行為標籤的冷啟動**：新用戶沒有行為資料怎麼辦？→ 預設給 neutral 權重 0.5，前 7 天不套疲勞度上限
2. **跨標籤權重衝突**：同時擁有「潛水型」+「活動揪團傾向」如何推？→ 需要定義**主導標籤**決策樹，不是所有標籤平權
3. **標籤暴漲治理**：IP 太多會造成推播過度分眾 → 需定義 Top-N 截斷或長尾合併規則
