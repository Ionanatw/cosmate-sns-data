在設計針對「投票（poll）」格式的推播時，為確保能有效喚活潛水用戶而不造成干擾，您可以結合系統資料庫欄位與標籤架構的動態調整機制，採取以下 4 個關鍵設計：

**1\. 尊重底層的「內向模式（Introvert Mode）」開關** 根據您的會員資料庫規格，用戶主檔（`cosmate_users`）中已經內建了 `introvert_mode`（休眠狀態：0為關閉，1為開啟）的欄位。在觸發輕量投票推播前，API 必須將開啟內向模式的用戶排除在推播名單之外，確保系統完全尊重潛水用戶當下「不想被打擾」的社交邊界。

**2\. 導入推播反饋迴圈（Feedback Loop）與行為追蹤** 利用我們先前定義的互動行為狀態欄位，追蹤潛水用戶對投票推播的真實反應。如果系統向潛水用戶發送了投票推播，但其回傳的 `action_type` 連續多次為忽略或滑掉（dismissed / ignored），系統必須將此視為明確的「負向反饋」，並透過演算法自動調降該用戶身上的「輕量互動/愛投票」標籤權重。

**3\. 設定嚴格的推播疲勞度控管與動態更新** 標籤架構必須具備實用性與動態更新的能力，避免過度依賴靜態標籤導致數據雜亂或推播疲勞。對於平時極少發文的潛水用戶，系統應設定更嚴格的推播上限（例如每週只推送 1-2 則與其核心 IP 最相關的投票）。若用戶長期未對這些低門檻的推播產生互動，系統應動態拔除其相關興趣標籤，停止無效推播。

**4\. 執行推播前的「絕對狀態」底層過濾** 在進行名單交集比對時，切記要在 API 閘道端加上帳號狀態的過濾機制。務必確認接收者的 `account_status` 仍為 `active`（正常），避免系統浪費運算資源，或將推播發送給狀態已被標記為 `deleted` 的無效帳號。
