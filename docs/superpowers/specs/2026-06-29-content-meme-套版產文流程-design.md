# content-meme 套版產文流程 — 設計文件

> 日期：2026-06-29｜主導對話串：德德（Claude Code）
> 範圍：CosMate Threads 產文流程上游重新規劃（選題 + 產文品質）
> 不在範圍：發布鏈核心、Metrics Sync、資料模型大改（僅 Telegram bot 審核卡需擴充）

---

## 1. 背景與問題

現行 `content-gen` 採「自由創作」模式：讀素材後生成全新貼文。鴿王回報產出**品質不穩定、無法直接用**，最高頻的兩個症狀是：

- **方向錯（根因）**：素材選錯角度，根基就歪，潤稿救不回
- **內容空泛/假**：為湊字數編造不存在的經歷，缺真實感

根因診斷：自由創作讓 AI「另起爐灶」，偏離已驗證會紅的素材結構，並在缺真實血肉時靠捏造填空。

## 2. 核心轉向：套版（transplant），不自由創作

把產文從「generative」改為「**套版**」——以市場已驗證的爆文**原文為骨架照抄**，只換人設皮。爆文骨架已驗證會紅 → 方向不會歪；血肉直接來自真實原文 → 不會假。同時殺掉上述兩個症狀。

新增獨立 skill **content-meme** 作為套版引擎（架構方案 B），與既有 `grab-idea`、`content-gen` 透過 Ideas DB 鬆耦合對接，不互相侵入。

## 3. 設計決策清單

| # | 決策 |
|---|---|
| D1 | content-gen 模式 → **套版**（爆文原文當骨架），由新 skill content-meme 實作 |
| D2 | 高保真：字大致照抄、只換關鍵詞，變動率 **15–25%** |
| D3 | 一筆素材最多給 **2–3 個人設**用（廢除交友類 union 全員 6 人） |
| D4 | 跨人設使用同一素材，發布間隔 **≥7 天**（防同框撞臉） |
| D5 | grab-idea 對 URL-only 素材**自動補抓原文全文**；抓不到 → 標「待補原文、不可套版」 |
| D6 | 缺完整原文的素材，content-meme **不可挑用** |
| D7 | 架構走**方案 B**：新增獨立中間層 skill，命名 **content-meme** |
| D8 | content-meme **主力**、content-gen **罕用備援**（無爆文可套時退回）；未來若套版流程不順，過渡到雙線並行 + 規則路由（演進路徑，現在不建） |
| D9 | **第二源頭**：daily/weekly 爬到的 **A 型爆款自動升級進 Ideas DB**（帶原文 + 建議 2-3 人設 + 自動視為可套版） |
| D10 | content-meme **預寫 2-3 人設版本** → Telegram 並排審核卡 → 鴿王點人設＝選那一版 → 發。沿用既有 Telegram 審核 bot，加「人設選擇鈕」 |
| D11 | `可套版` 不新增欄位，由 **Ideas DB「Slack note」非空自動推導**（schema 零改動） |
| D12 | 「自動發佈」定義 = **自動產文 + 推 Telegram 等鴿王點**，無完全無人工的直發路徑 |
| D13 | （2026-06-29 長文樣本驗證新增）content-meme **只做逐字套版**（爆點在字句、內容留同域可套）；**格式套版**（爆點在格式、內容須換域重寫，如清單/接龍/問答模板）**路由給 content-gen**。判斷訊號：一篇怎麼改都進不了 15–25%（同域輕改 too_low 撞臉、一改就 too_high 換域），即非逐字套版料 → 丟 content-gen。 |

## 4. 端到端流程

```
源 1: Slack #sns-viral-idea → [grab-idea 強化]
        • URL-only → 自動爬原文全文存進 Slack note；抓不到 → 留空（=不可套版）
        • AI摘要升級為「拆出可複製結構」(Hook型/段落骨架/爆點)
        • 指派 2-3 最適人設（廢 union 6 人）
源 2: 每日/每週爬取 → [daily.sh / weekly.sh 末端新增升級步驟]
        • 篩 Type=A(全能爆款) + zh-TW + 有原文 → 自動寫進 Ideas DB
        • 帶原文(Slack note) + Priority=High｜可寫 + 依主題建議 2-3 人設 + 來源=🔥爬取爆款
        • 寫入前比對既有原文去重
                      │
                      ▼
            ┌──────────────────┐
            │  Ideas DB 可套版池  │  Slack note 非空 + 2-3 人設 + Priority=High｜可寫
            └─────────┬────────┘
                      │  ★ [content-meme]
                      │   為短名單 2-3 人設「各寫一版」套版稿
                      │   (骨架忠實、變動 15-25%、套版 QC、防撞臉查詢)
                      ▼
            ┌──────────────────┐
            │  Telegram 審核卡    │  並排 2-3 版 + 人設選擇鈕（每鈕=一版）
            └─────────┬────────┘
                      │  鴿王點某人設 = 選那一版
                      ▼
            選中版寫入 Posts DB → Threads API 發布（對應帳號）
                      │  防撞臉守則：同素材跨人設 ≥7 天、≤3 人設
                      ▼
            回寫 Notion → Metrics Sync（既有，不動）
```

## 5. content-meme 規格（核心）

### 5.1 進場條件
content-meme 只挑符合以下全部的 Ideas DB 列：
```
Priority   = High｜可寫
Slack note ≠ 空           ← 推導「可套版」(D11)
貼文人     ∋ 短名單人設    （2-3 人設之一）
防撞臉檢查 通過            （見 5.5）
```

### 5.2 轉寫流程（6 步）
```
1. 載入完整原文（Ideas DB Slack note，一字不差）
2. 載入目標人設 Pillar Description（口吻）+ 近 3-5 篇貼文（校準）
3. 結構解析：拆原文成骨架 → Hook句型 / 段落序與功能 / 金句 / 互動結尾型 / 節奏
4. 套版換皮（15-25% 變動，見 5.3）
5. 套版 QC（見 5.4）
6. 為短名單每個人設各產一版 → 推 Telegram 並排卡
```
鴿王在 Telegram 點某人設後，選中版才寫入 Posts DB（Status=Cooking、審核狀態=⏳待審核、is_trending_based=true、來源=🤖AI初稿），並回填 relation：Ideas.應用貼文 ↔ Posts。未選的版本捨棄。

### 5.3 ★ 15–25% 換皮配方（整個 skill 的心臟）

| 保留 75-85%（骨架照抄） | 替換 15-25%（換人設皮） |
|---|---|
| 開頭 Hook 的句型與鉤法 | Hook 裡的專有名詞 / 情境 → 換進人設世界 |
| 段落數、每段功能、順序 | 主角第一人稱身份 → 換成該人設 |
| 金句 / 爆點句的結構 | 品牌或產品名 → 換掉或移除（避業配感） |
| 列表 / 蓋樓 / 問答的格式 | 結尾互動 CTA → 改成人設語氣 |
| 整體節奏與長度 | Hashtags → 人設 / 主題標籤 |

口吻僅做**輕微**調整（非重寫），骨架忠實優先。

### 5.4 套版 QC（從「夠不夠創意」改成「夠不夠忠實 + 不撞臉」）

**新增核心檢查：**
- **QC-保真**：成品骨架與原文對齊（段落數 / Hook型 / 結尾型一致）
- **QC-語義一致**（2026-06-29 新增，樣本驗證踩出的洞）：換入的專有名詞，彼此間必須維持原文那種**真實世界關係**（音樂人↔他演唱的作品、作品↔角色）。**禁止盲換名詞**——把不相干的詞硬塞進原文的關係句式，會產生「看似有依據、其實是假」的內容，比自由創作更危險（披著「抄了真爆文」的外皮，更易矇混過 QC 後被圈內人抓包）。無法查證真實關係就別動那個位置，必要時 web search 查證後再換。**語義誠實優先於變動率**：若維持真實關係導致變動率 >25%，是「這篇不適合套給這個人設」的訊號，換素材/換人設，不為壓變動率而造假。
- **QC-變動率**：實際變動落在 **15–25%**。太低→抄太死有侵權/撞臉風險；太高→失去套版意義變回創作。← 新核心閘門。**只量「改了多少」、量不出「改得對不對」，過了不代表 QC-語義一致 過。**
- **QC-身份**：第一人稱設定不違反人設身份（動漫宅 Olie 不能冒出不符身份的經歷）
- **QC-撞臉**：與同 idea 其他人設版本、及該人設近期貼文，相似度不過高

**沿用舊 QC（發布契約，不能丟）：**
- 正體中文零簡體
- 禁用詞 + 高頻錯字（不二錯 DB）
- 業配合規（真人經歷不提產品名）
- **heading 5 段結構完整**（telegram-bot fetchPostContent 抓內文靠這個，不二錯 #063）
- 字數限制

### 5.5 防撞臉守則（產文前必跑）
```
對某 idea 要產給人設 P 之前，查 Ideas.應用貼文 relation：
  • 已關聯貼文的「貼文人」去重 → 若已達 3 個人設 → 不再產第 4 個
  • 跨人設的 Post date 必須間隔 ≥7 天 → 7 天內已有別人設用過 → 排到 ≥7 天後或提示鴿王
```
靠現有 Ideas↔Posts 雙向 relation 查詢，**不需新欄位**。

## 6. 上游與相依改動

### 6.1 grab-idea 強化（源 1）
- 新增「URL-only → 自動爬原文全文」存進 Slack note；抓不到 → 留空（=不可套版）
- AI摘要從「一句摘要」升級為**拆出可複製結構**，給 content-meme 當藍圖
- 指派 **2-3 最適人設**，廢除「交友類 union 全員 6 人」

### 6.2 自動升級橋（源 2，新增）
- 在 `daily.sh` / `weekly.sh` 末端加一步：篩 `Type=A + zh-TW + 有原文` → 寫進 Ideas DB
- 自動帶：原文（Slack note）、Priority=High｜可寫、依主題建議 2-3 人設、來源=🔥爬取爆款
- 主題→人設對應：動漫→Olie；交友→Dadana/Amy/Kiki；Cosplay→CosMate/Dadana
- 寫入前比對 Ideas DB 既有原文去重

### 6.3 Telegram bot 擴充（德德 VPS，實作相依項）
- 沿用既有 `/掃描` 審核卡 + 互動按鈕 pipeline
- 新增：一張卡並排顯示同一 idea 的 2-3 人設版本 + 一排人設選擇鈕
- 鴿王點某人設 → 該版寫入 Posts DB 並進入既有發布流程
- 注意：此為超出 content-meme/grab-idea 兩個 skill 的程式改動

### 6.4 content-gen 去留
- 標記 **deprecated-but-kept**，僅「想發某主題但無爆文可套」時退回使用
- 演進路徑：若套版流程不順 → 過渡到雙線並行 + 規則路由（未來再建）

### 6.5 資料模型
- **零 schema 改動**。`可套版` 由 Ideas DB「Slack note」非空自動推導（D11）
- 沿用 Posts DB `is_trending_based`、`來源`、`Status`、`審核狀態` 等既有欄位

## 7. 驗證方式
1. 取近期 3-5 篇 A 型爆款，手動跑一遍 content-meme，逐項檢查：變動率落在 15-25%？骨架忠實？不撞臉？人設身份對？
2. 人工驗證通過後，再接通自動升級橋（源 2）與 Telegram 審核卡擴充
3. 灰度：先單一人設、單一主題試跑一週，看實際發布成效（Metrics Sync 數據）再放大

## 8. 已知風險
- **高保真照抄的撞臉/侵權殘餘風險**：鴿王已知情並以「≤3 人設 + ≥7 天錯開」承擔（D3/D4）
- **「Slack note 非空」推導可能誤判**：若 note 只有片段而非完整原文，會被誤判為可套版。緩解：grab-idea 抓不到完整原文時必須留空 note，不存片段
- **Telegram bot 改動跨出 skill 邊界**：需德德在 VPS 端同步部署，屬實作相依，非單純 skill 修改
- **A 型自動入庫的品質**：A 型門檻嚴（讚和留言均 ≥P90），但仍可能混入不適合 CosMate 調性的題材；靠下游 content-meme 產文 + Telegram 審核兩道人工關把守
